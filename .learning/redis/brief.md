# S1 브리프 — 오픈 직전 조회 폭주

## 상황
플래시 세일 오픈 60초 전. 유저들이 같은 상품 상세 페이지를 새로고침 폭격 중이다.
이 순간 상품 정보는 거의 안 바뀐다 — 트래픽만 몰린다.
현재 서비스는 이 부하에서 **p99 = 173ms** (100 VU, 30초, 1,265 req/s). 유저는 100ms 넘게 걸리면 이탈하기 시작한다.

## 목표 (행동 계약 — `k6/s1-load.js`)
- 부하 속 **모든 응답이 정확**하고 (checks 100%)
- **p99 < 100ms** (코치 PC 숫자 아님 — **네 머신에서 잰** naive 베이스라인 173ms 대비 설정한 값이다)

어떻게 달성할지는 지정하지 않는다. 진단도 해법 선택도 네 몫이다.
SLO 숫자가 네 머신 기준 불합리하면 데이터 들고 반박해라 — 같이 조정한다.

## 실행 방법
```powershell
# 1. 서버 띄우기 (호스트)
./gradlew bootRun
# 2. 다른 터미널에서 부하 발사 (30초)
docker run --rm -i grafana/k6 run - < k6/s1-load.js
```
합격/불합격은 k6가 마지막에 THRESHOLDS로 판정해준다.

## 측정 (진단 — 합격 조건 아님)
- k6 출력: avg / med / p90 / p95 / p99 / max, req/s, 실패율
- 서버 로그, Hikari 풀 상태 등 네가 보고 싶은 것 무엇이든
- DB에 실제로 요청이 얼마나 오는지는 DB 쪽에서 직접 셀 수 있다 — MySQL은 자체 카운터를 갖고 있다 (`SHOW GLOBAL STATUS`)
- 주의: 측정 자체를 오염시키는 설정이 서버에 하나 켜져 있다. 의심해봐라.

## knob
`k6/s1-load.js`의 `vus` — 네 fix가 green이 되면 코치가 이걸 올려 한계를 같이 확인한다.
compose 불변, `--scale app=1` 유지.

## 멈춤
green을 만들었거나, 진단 숫자를 들고 막혔을 때 → 코치 호출.

---
### 측정 도구 이력 (왜 바뀌었나)
- JUnit 내장 부하(폐기): 요청 쏘는 쪽과 받는 쪽이 같은 JVM/CPU를 나눠 써서 잰 시간이 오염됨.
  같은 naive 서버를 JUnit은 p99=408ms로, k6는 173ms로 쟀다 — 차이(2.3배)가 곧 측정기 자신의 잡음이었다.
- k6(도커, 현행): 서버 밖에서 손님 관점으로 잼. 이미지로 도구 버전 고정 → 어디서든 같은 도구.
